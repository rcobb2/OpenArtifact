# OpenArtifact

A self-hosted, local-first vault for HTML and React/TSX artifacts generated by AI assistants (Claude, ChatGPT, Gemini, …). Think **Obsidian for artifacts**: they live as plain `.html`/`.tsx` files on disk that you fully own, with a SQLite index on top for search, tags, and notes.

## Quick start (Docker)

```bash
docker compose up -d
# → open http://localhost:4747
```

Or without compose:

```bash
docker build -t openartifact .
docker run -d -p 4747:4747 -v "$(pwd)/data:/data" --name openartifact openartifact
```

Your artifacts land in `./data/artifacts/*.html` (human-readable, one file per artifact) with metadata in `./data/index.db`. Back up or sync that folder however you like — the files stand alone.

## Quick start (plain Node, no Docker)

Requires Node ≥ 22.13 (uses the built-in `node:sqlite` — zero npm dependencies, nothing to install):

```bash
node server.js
# → http://localhost:4747
```

Optional env vars: `PORT` (default 4747), `DATA_DIR` (default `./data`), `HOST`.

## Importing artifacts

Import is designed to be frictionless:

1. **Drag & drop** one or more `.html` files anywhere on the page.
2. **Paste** raw HTML anywhere on the page (Ctrl/⌘+V) or into the paste box — imports instantly. Copy an artifact's source from Claude/ChatGPT, click into OpenArtifact, paste, done.
3. **Upload** via the file picker button.
4. **Import from URL** — paste a published artifact share link or any page URL.
5. **CLI/API** — one universal endpoint, `POST /api/import`, takes a file, a URL, or raw HTML:

   ```bash
   # upload a file (or several)
   curl -F "file=@artifact.html" http://localhost:4747/api/import
   curl -F "file=@a.html" -F "file=@b.html" -F "folder=Work/Dashboards" -F "tags=api,batch" \
        http://localhost:4747/api/import

   # import from a URL
   curl -X POST -H 'Content-Type: application/json' \
        -d '{"url":"https://example.com/artifact", "folder":"Work"}' http://localhost:4747/api/import

   # pipe raw HTML
   curl -X POST -H 'Content-Type: text/html' --data-binary @artifact.html http://localhost:4747/api/import
   ```

   `folder` accepts a path like `Work/Dashboards` — missing folders are created automatically (or pass an exact `folder_id`). Optional `title`, `tags`, `notes` fields work in all forms. Responds `201` with `{imported: […], errors: […]}`.

Titles are auto-extracted from `<title>` (or first `<h1>`). Edit title, tags, and notes from the detail view.

## Library features

Gallery with live sandboxed previews, full-text search over titles/tags/notes, tag filter chips, full-screen render, metadata editing, delete, and download of the original file. Three sample artifacts (and a sample folder tree) are seeded on first run — delete them freely.

## Folders

The sidebar holds a nested, Obsidian-style folder tree. The homepage shows **Unsorted** — artifacts not assigned to any folder; clicking a folder shows its artifacts. Create folders with the ＋ button (or the ＋ on a folder row for a subfolder); rename (✎) and delete (✕) from the row's hover actions. Deleting a folder removes it and its subfolders, but never deletes artifacts — anything inside moves back to Unsorted.

Move artifacts by dragging a card onto a folder in the sidebar, or via the Folder dropdown in the detail view. New imports (drop/paste/upload/URL) land in whichever folder you're currently viewing. Search always spans all folders; results show each hit's folder path.

**On-disk choice:** folders are real subdirectories under `data/artifacts/`, named exactly as you named them, so the vault stays human-browsable in Finder/Explorer (e.g. `data/artifacts/Samples/Generative/particle-field-….html`). Renaming or moving a folder, or moving an artifact, moves the actual files/directories on disk. The SQLite index stores the hierarchy (`folders` table + `folder_id` on artifacts); existing pre-folder databases are migrated automatically on startup. One consequence: edit the tree through the app (UI or API), not by shuffling files by hand — the index won't see manual moves.

## API

| Method | Path | Description |
|---|---|---|
| GET | `/api/artifacts?q=&tag=&folder=` | List/search. `folder` = a folder id or `root` (unsorted); `q` searches across all folders |
| POST | `/api/import` | **Universal import**: multipart file upload(s), JSON `{url}` or `{html}`, or raw `text/html` body; optional `title`, `tags`, `notes`, `folder_id` or `folder` path (auto-created) |
| POST | `/api/artifacts` | Import (JSON `{html,title?,tags?,notes?,folder_id?}` or raw `text/html` body) |
| POST | `/api/import-url` | Import from URL (`{"url":"https://…","folder_id?":…}`) — kept for back-compat; prefer `/api/import` |
| GET/PATCH/DELETE | `/api/artifacts/:id` | Get / edit metadata (incl. `folder_id` to move) / delete |
| GET | `/api/tags` | Tag counts |
| GET | `/api/folders` | Folder tree with artifact counts |
| POST | `/api/folders` | Create folder (`{"name":…, "parent_id?":…}`) |
| PATCH/DELETE | `/api/folders/:id` | Rename/re-parent (`{name?, parent_id?}`) / delete (contents move to Unsorted) |
| GET | `/raw/:id` | Render artifact (`?download=1` to download) |

## React/TSX artifacts

The vault accepts `.tsx`/`.jsx` React components the same way it accepts HTML — drop the file, paste the source, upload via API (`"kind": "tsx"` in JSON, or just use a `.tsx` filename in multipart; pasted/posted TSX is also auto-detected). The component source is stored verbatim on disk as a `.tsx` file.

Rendering happens in the same sandboxed iframe, via a wrapper page that transpiles the source in-browser (Babel standalone), resolves bare imports from [esm.sh](https://esm.sh) pinned to a single React 18, loads Tailwind (most AI-generated components assume it), and mounts the default export. Consequences: TSX **previews need internet access** at view time (the .tsx files themselves remain fully yours and offline-safe on disk), the component must have an `export default`, and imports must be npm packages esm.sh can serve (react, recharts, lucide-react, d3, lodash, … — fine for typical Claude artifacts; relative imports to other files won't resolve). `?download=1` always returns the original untouched source.

## Claude skill: auto-save artifacts to the vault

`skill/save-to-openartifact/` contains a Claude skill that makes Claude automatically POST every completed HTML artifact it generates to this vault via `/api/import` (titled, tagged `claude`, filed under a `Claude/<Topic>` folder, with replace-on-revision to avoid duplicates). Install it from the packaged `.skill` file (Claude → Settings → Capabilities → Skills) or copy the folder into your skills directory.

Reach-the-vault caveat: Claude Code, Cowork, and other locally-running environments can hit `http://localhost:4747` directly. claude.ai web chat (Chat/Projects/Dispatch) executes server-side and cannot reach your localhost — point the skill's `VAULT_URL` at a secure tunnel (Tailscale Funnel, Cloudflare Tunnel) if you want auto-save from the web app, and put auth in front before exposing the vault.

## Security model

Artifacts are untrusted code. They render only inside `<iframe sandbox="allow-scripts">` (no `allow-same-origin`, so they run in an opaque origin with no access to the app, your cookies, or other artifacts). The `/raw/:id` endpoint additionally serves a `Content-Security-Policy: sandbox` header, so an artifact opened directly in a tab is still origin-isolated. The URL-import feature fetches whatever URL you give it from inside the container — it's intended for a trusted, local, single-user deployment; don't expose the port to the internet without adding auth.

## Layout

```
server.js          # entire backend (~400 lines, zero dependencies)
public/index.html  # entire frontend (single file)
seed/              # sample artifacts imported on first run
data/              # created at runtime: artifacts/<folders>/*.html + index.db
```

Hack away — there's no build step.
